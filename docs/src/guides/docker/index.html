BSB

Docker

Run the BSB container and provide plugins at startup. BSB is container-first in production; plugin projects are built separately and mounted or installed into the container.

Image Tags

Use the runtime-prefixed tags. Forgejo is the primary registry; Docker Hub is a best-effort compatibility mirror and can lag if Docker Hub rejects a push. For Node.js production deployments, use:

code.bettercorp.dev/bettercorp/service-base:node
code.bettercorp.dev/bettercorp/service-base:node-latest
code.bettercorp.dev/bettercorp/service-base:node-9.5.1
Do not use tags like 9-node. BSB tags are runtime-first: node, node-latest, or node-<version>. The same pattern applies to other runtimes, such as go, python, and dotnet.

Two Docker Workflows

BSB has two separate Docker concerns:

  • Build service images: package your service/plugin code with bsb-plugin-cli build, then create an image that extends the BSB runtime image or ships a prepared plugin directory.
  • Deploy service containers: run the BSB image with runtime config, writable paths, and any extra plugin packages or mounted plugin directories.

Do not use a plain tsc, node script.js, or ts-node command as the service build/start path. BSB plugins should build through the BSB plugin CLI and run through the BSB lifecycle.

Bundled Node.js Plugins

The official Node.js runtime image runs BSB from /home/bsb/node_modules/@bsb/base, and built-in core plugins resolve from that package. The runtime package includes npm run start, npm run debug, and npm run plugins for shell/debug use inside the container. It also includes @bsb/config-vault in /home/bsb/node_modules. Reference Vault plugins directly by package name in your config:

service-config-vault:
  plugin: service-config-vault
  package: "@bsb/config-vault"
  enabled: true

Do not also add @bsb/config-vault to BSB_PLUGINS when using the official Node image. It is already installed with the image, like the base package. Use BSB_PLUGINS only for additional published plugins that are not bundled, and pin those versions.

Build Service Images

Use this flow when you are producing an application/service image. The service image should contain already-built plugin artifacts, not raw TypeScript that compiles when the container starts.

FROM code.bettercorp.dev/bettercorp/service-base:node

WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Optional: verify what BSB can discover in the built image.
RUN node /home/bsb/node_modules/@bsb/base/lib/scripts/list-plugin-search-paths.js

The project npm run build should call bsb-plugin-cli build so schemas, generated clients, hooks, validation, and assets are handled consistently. Avoid adding a direct tsc command in place of the BSB plugin build.

Build-Time Environment

  • The official Node image already sets NODE_ENV=production, BSB_CONTAINER=true, BSB_LIVE=true, and BSB_PLUGIN_DIRS=/mnt/plugins. Derived Dockerfiles do not need to restate those defaults.
  • BSB_SHOW_PACKAGES=true is a runtime switch. During a Docker build, call node /home/bsb/node_modules/@bsb/base/lib/scripts/list-plugin-search-paths.js directly after installing or copying plugins.
  • If the build needs private npm packages, configure npm credentials at build time using your CI secrets. Do not bake tokens into the final runtime image.

Deploy Service Containers

Deployment is where you choose the active config plugin, config source, writable volumes, and any extra plugins installed at startup. The container entrypoint prepares writable paths, optionally installs requested plugin packages, prints the optional package report, and then starts BSB.

Run With Mounted Plugins

Mount prebuilt plugin packages and point BSB at the plugin directory. The first path in BSB_PLUGIN_DIRS is also used as the install target when BSB_PLUGINS is set.

docker run --rm \
  -v "$(pwd)/plugins:/mnt/plugins:ro" \
  -v "$(pwd)/sec-config.yaml:/app/sec-config.yaml:ro" \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  -e BSB_CONFIG_FILE=/app/sec-config.yaml \
  code.bettercorp.dev/bettercorp/service-base:node

Install Plugins On Startup

Use BSB_PLUGINS when the container should install published plugin packages before starting. Supported selectors are no selector, major-only, minor, or exact versions, such as @scope/plugin@10, @scope/plugin@10.2, and @scope/plugin@10.2.3. Pin exact versions for the most deterministic production boots; use major or minor selectors only when controlled updates are intended. @latest is rejected.

docker run --rm \
  -v bsb-plugins:/mnt/plugins \
  -v "$(pwd)/sec-config.yaml:/app/sec-config.yaml:ro" \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  -e BSB_PLUGINS='@myorg/service-api@1.2.3,@myorg/observable-custom@2.4.1' \
  -e BSB_CONFIG_FILE=/app/sec-config.yaml \
  code.bettercorp.dev/bettercorp/service-base:node-latest

Compose

services:
  bsb:
    image: code.bettercorp.dev/bettercorp/service-base:node
    environment:
      BSB_PLUGIN_DIRS: /mnt/plugins
      BSB_CONFIG_FILE: /app/sec-config.yaml
      BSB_WRITABLE_PATHS: /data
    volumes:
      - ./plugins:/mnt/plugins:ro
      - ./sec-config.yaml:/app/sec-config.yaml:ro
      - bsb-data:/data

Config Sources

BSB loads exactly one active config plugin. The active config plugin is selected before normal observable, events, and service plugins are loaded.

YAML File Config

Use config-default for file-based deployments. This is the default path when no config plugin override is set.

environment:
  BSB_CONFIG_PLUGIN: config-default
  BSB_CONFIG_PLUGIN_PACKAGE: "@bsb/base"
  BSB_CONFIG_FILE: /app/sec-config.yaml
  BSB_PROFILE: default

Full JSON Env Config

Use config-env when the whole BSB config is supplied as one JSON environment value. This avoids mounting sec-config.yaml.

environment:
  BSB_CONFIG_PLUGIN: config-env
  BSB_CONFIG_PLUGIN_PACKAGE: "@bsb/base"
  BSB_PROFILE: default
  BSB_CONFIG_JSON: >
    {"default":{"observable":{},"events":{},"services":{}}}

Vault Config

Use config-vault for containers that fetch their published runtime config from Vault. Runtime service containers do not define application, service group, or deployment profile in YAML; the Vault API key is bound to those server-side.

environment:
  BSB_CONFIG_PLUGIN: config-vault
  BSB_CONFIG_PLUGIN_PACKAGE: "@bsb/config-vault"
  vaultUrl: https://vault.example.com
  apiKeyId: vk_xxx
  apiSecret: vs_xxx

Config plugin schema keys are passed as environment variables using their exact names. For Vault, that means lower camel case keys such as vaultUrl, apiKeyId, apiSecret, timeoutMs, and allowInsecureHttp.

Vault Service Deployment

The Vault manager service is different from a runtime service that consumes Vault. The manager service can use config-env to bootstrap service-config-vault itself:

services:
  postgres:
    image: postgres:18-alpine

  vault:
    image: code.bettercorp.dev/bettercorp/service-base:node-latest
    environment:
      BSB_CONFIG_PLUGIN: config-env
      BSB_CONFIG_PLUGIN_PACKAGE: "@bsb/base"
      BSB_PROFILE: default
      BSB_CONFIG_JSON: >
        {"default":{"observable":{},"events":{},"services":{"service-config-vault":{"plugin":"service-config-vault","package":"@bsb/config-vault","enabled":true,"config":{"host":"0.0.0.0","port":8080,"publicUrl":"https://vault.example.com","production":true,"databaseUrl":"postgres://vault:secret@postgres:5432/bsb_vault","masterKey":"BASE64_32_BYTE_KEY"}}}}}

Environment Variables

The official Node image already sets NODE_ENV=production, BSB_CONTAINER=true, BSB_LIVE=true, and BSB_PLUGIN_DIRS=/mnt/plugins. Dockerfiles that extend code.bettercorp.dev/bettercorp/service-base:node or code.bettercorp.dev/bettercorp/service-base:node-latest do not need to repeat those defaults unless they intentionally override them.

  • BSB_PLUGIN_DIRS: comma-separated plugin directories. Prefer this over legacy aliases.
  • BSB_PLUGINS: comma-separated npm packages to install at container start. Supports no selector, major, minor, and exact selectors. If installation fails, the container exits before BSB starts.
  • BSB_PLUGIN_UPDATE=true: refresh already-installed plugins on startup.
  • BSB_WRITABLE_PATHS: comma-separated files or directories that the entrypoint should create if needed, chown to the Node user, and chmod as writable. Use this for runtime state such as /data.
  • BSB_CONFIG_PLUGIN: active config plugin id, such as config-default, config-env, or config-vault.
  • BSB_CONFIG_PLUGIN_PACKAGE: package that contains the config plugin, such as @bsb/base or @bsb/config-vault.
  • BSB_CONFIG_FILE: config file path inside the container.
  • BSB_CONFIG_JSON: full runtime config JSON for config-env.
  • BSB_PROFILE: config profile to load, defaulting to default.
  • vaultUrl, apiKeyId, apiSecret: required runtime Vault config plugin values when BSB_CONFIG_PLUGIN=config-vault.
  • timeoutMs, allowInsecureHttp: optional Vault config plugin values.
  • BSB_SHOW_PACKAGES=true: print the package/plugin discovery report before BSB starts, including package names, versions, plugin types, paths, and config-reference snippets.
  • BSB_DEBUG=true: enable production debug logging.

Build Plugins Separately

Plugin projects should build with bsb-plugin-cli build before deployment. The container loads built plugin packages; it should not compile application plugin source at startup.

Build-Time Plugin Discovery

The Node build logs a package/plugin search report automatically. For Dockerfiles that extend the BSB Node image and add plugins in a later build stage, run the same report after copying or installing those plugins:

RUN node /home/bsb/node_modules/@bsb/base/lib/scripts/list-plugin-search-paths.js

At runtime, enable the same report with BSB_SHOW_PACKAGES=true.

When working in this repository, the equivalent command is:

npm run list-plugin-search-paths --workspace nodejs
