Architecture

How BSB is designed: components, processes, and flows.

Core Principles

Event-Driven

All communication is via events on a pluggable event bus. Services emit, listen, and perform returnable calls without tight coupling.

Plugin Architecture

Functionality is delivered through plugins (configuration, events, logging, metrics, services). The container discovers and loads prebuilt plugins.

Container-First

BSB runs as a container in production. Your deployable unit is the container plus a mounted plugin repository.

High-Level Components

BSB Runtime

  • Core Lifecycle, plugin loader, health
  • Events Event bus abstraction and adapters
  • Logging Structured logs with pluggable sinks
  • Metrics Counters, gauges, histograms, traces
  • Config Typed configuration with sources and profiles

Plugin Repository

Mounted directory containing organizations and versions.

/mnt/plugins/ @org/service-foo/ latest/ 1.2.3/ @org/events-rabbitmq/ latest/

Language-specific artifacts; see language guides for format.

Lifecycle: Boot to Ready

  1. Bootstrap: Load configuration profile (BSB_PROFILE) and core adapters (events/logging/metrics/config).
  2. Discover: Scan BSB_PLUGIN_DIR for organizations and versions; resolve desired plugin set.
  3. Validate: Check compatibility, config schemas, and dependencies; order plugins.
  4. Initialize: Instantiate plugins; provide context: logging, metrics, config, and event handles.
  5. Register Events: Each plugin registers listeners and returnable handlers.
  6. Run: Start plugins; expose health/readiness; begin handling traffic.

Event Communication

Event Types

KindDescription
Fire-and-forgetEmit without waiting for response.
Returnable (RPC-like)Emit and await a typed response.
BroadcastEmit to many subscribers.

Tracing & Observability

  • Every event carries a trace context (trace/span identifiers)
  • Metrics record latencies, rates, and errors
  • Structured logs correlate to traces and events

Configuration & Profiles

Profiles

  • Profiles select environment-specific settings (BSB_PROFILE)
  • Layered sources (files, vaults, services) merged into a typed config

Secrets & External Sources

Use configuration plugins to pull from vaults (e.g., 1Password), service discovery (Consul/etcd), or orchestrator-native stores.

Resilience & Scaling

Resilience

  • Design for backpressure and timeouts at the event layer
  • Implement retries/jitter and circuit breaking in your plugin code or chosen libraries (framework-agnostic)
  • Prefer idempotent handlers to simplify failure recovery

Scaling

  • Horizontal scale via multiple containers/services
  • Per-service autoscaling based on metrics
  • Edge placement for low-latency flows

Security Considerations

  • Signed plugin artifacts and checksum verification (pipeline responsibility)
  • Least-privilege configuration sources and secret handling
  • Audit logs on plugin load/unload and configuration changes

Operational Flows

Deploy

  1. Build container image
  2. Publish plugins to repository
  3. Configure BSB_PLUGIN_DIR and profile
  4. Roll out

Upgrade Plugin

  1. Publish new version under @org/name/x.y.z
  2. Switch alias (e.g., latest) or pin specific version
  3. Restart/roll services to pick up

Roll Back

  1. Repoint alias to prior version
  2. Roll services
  3. Monitor metrics and logs
