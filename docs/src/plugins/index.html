Plugin System

Discover and use plugins to extend BSB functionality

Plugin Architecture

BSB's plugin system extends configuration, events, logging, metrics, and services. In production, run the BSB container and supply prebuilt plugins via a mounted repository (BSB_PLUGIN_DIR). For local development, use {src|lib}/plugins/ in a plugin project for types and testing. Plugins are discovered automatically based on configuration.

🛠️ Developing Custom Plugins

Create your own plugins to extend BSB. In production, the BSB container loads prebuilt plugins from a mounted repository specified by BSB_PLUGIN_DIR. The repository is organized by organization and version (for example, @org/plugin-name/latest and semantic versions). The exact artifact layout and manifests are language-specific. See language guides for details.

Repository Layout (Generic)

/mnt/plugins/
  @org/plugin-one/
    latest/
      
    1.0.0/
      
  @org/plugin-two/
    latest/

Naming and Categories

  • Prefix names with their category when possible: config-*, events-*, logging-*, metrics-*, service-*
  • Use lowercase kebab-case names for consistency
  • Each plugin is self-contained and defines its capabilities via typed events and configuration

Language-specific Implementation

For packaging, manifests, and code templates, see the language guides:

  • Node.js plugin implementation

Plugin Discovery

The container discovers and loads plugins by:

  • Scanning the mounted BSB_PLUGIN_DIR for organizations and available versions
  • Reading language-specific manifests/metadata for compatibility and configuration
  • Resolving the desired set from configuration (e.g., default set plus overrides)
  • Initializing plugins in dependency-aware order
