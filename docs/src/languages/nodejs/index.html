Node.js Implementation

Complete guide to building BSB plugins with Node.js and typed event interfaces

Requirements

  • Node.js >= 23.0.0
  • npm >= 11.0.0
  • Docker (runtime - production)
  • TypeScript & Node.js (optional - for plugin development)

Quick Installation (Dev)

For local plugin development and type support:

npm install @bsb/base

# Create your plugin structure (dev)
mkdir -p src/plugins/service-my-api
echo 'console.log("BSB service ready!");' > src/plugins/service-my-api/index.ts

How BSB Works

BSB is designed to run as a standalone Docker container that loads and runs plugins (published separately). In production you deploy the container and provide plugins via BSB_PLUGIN_DIRS (comma-separated) or BSB_PLUGINS. Installing BSB as a dependency is only for contributors and local plugin development.

BSB v9 Documentation

BSB v9 introduces the Observable pattern for unified observability. Check out our comprehensive guides:

  • Migration Guide: v8 to v9 - Step-by-step migration from DTrace to Observable
  • Observable Pattern Guide - Complete usage guide with examples
  • API Reference (TypeDoc) - Full TypeScript API documentation

Discovery & Runtime

  • Production: run the BSB container and provide plugins via BSB_PLUGIN_DIRS
  • Optional: specify explicit versions with BSB_PLUGINS (e.g. service-my-api@1.2.3;observable-default@latest)
  • Development: install BSB in your plugin project for types and run locally if needed

Project Structure

Container-first plugin repository layout (recommended for production):

/mnt/plugins/ @org/service-my-api/ latest/ package.json lib/plugins/service-my-api/index.js 1.2.3/ package.json lib/plugins/service-my-api/index.js @org/logging-custom/ latest/ package.json lib/plugins/logging-custom/index.js @org/metrics-prometheus/ latest/ package.json lib/plugins/metrics-prometheus/index.js

Plugin development project structure (local dev):

my-plugin/ |-- package.json |-- src/ | `-- plugins/ | `-- service-my-api/ | `-- index.ts `-- lib/ (built) `-- plugins/service-my-api/index.js

Container Quick Start

Run the official BSB container and mount your plugin repository:

docker run --rm \
  -v $(pwd)/plugins:/mnt/plugins:ro \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  betterweb/service-base:node
Note: This is the supported production deployment. Use comma-separated paths for multiple plugin directories.

    Build through BSB: Do not replace npm run build with tsc, and do not run compiled files directly with node lib/... or ts-node. The plugin CLI runs schema extraction, virtual client generation, hooks, TypeScript compilation, asset/schema copying, manifests, and the BSB lifecycle.

  2. Create Plugin Structure

    Set up the plugin directory structure:

    # Create plugin structure
mkdir -p src/plugins/service-my-api

    Keep each plugin self-contained. Put its schemas, assets, helpers, generated inputs, and local docs under its plugin project or plugin folder, then import/reference them from elsewhere when needed. Avoid shared side directories that another plugin must know about to build or run.

  3. Create Your First Service Plugin

    Implement a basic service plugin with typed events and the correct lifecycle (constructor -> init -> run -> dispose):

    import {
  BSBService,
  BSBServiceConstructor,
  Observable,
  bsb,
  createConfigSchema,
  createEventSchemas,
  createFireAndForgetEvent,
  createReturnableEvent,
  createBroadcastEvent,
} from '@bsb/base';
import * as av from 'anyvali';

// Plugin metadata + config schema (auto-generates bsb-plugin.json)
// BSB strips unknown config keys centrally during startup.
const Config = createConfigSchema(
  {
    name: 'service-my-api',
    description: 'My API service plugin',
    tags: ['service', 'api'],
  },
  av.object({}),
);

// Cross-language event schemas with BSB types
const EventSchemas = createEventSchemas({
  emitEvents: {
    'notification.send': createFireAndForgetEvent(
      bsb.object({
        type: bsb.string(), // could be a literal/enum if needed
        userId: bsb.uuid(),
        message: bsb.string({ optional: true }),
      }, 'Send notification to user'),
      'Send notification to user',
    ),
  },
  onEvents: {
    'user.created': createFireAndForgetEvent(
      bsb.object({
        userId: bsb.uuid(),
        email: bsb.string(),
      }, 'User created event'),
      'Handle user creation event',
    ),
  },
  emitReturnableEvents: {},
  onReturnableEvents: {
    'api.getUser': createReturnableEvent(
      bsb.object({ userId: bsb.uuid() }, 'Get user input'),
      bsb.object({
        userId: bsb.uuid(),
        status: bsb.string(),
      }, 'Get user output'),
      'Get user status information',
    ),
  },
  emitBroadcast: {
    'system.alert': createBroadcastEvent(
      bsb.object({
        level: bsb.string(),
        message: bsb.string(),
        timestamp: bsb.datetime(),
      }, 'System alert'),
      'System-wide alert broadcast',
    ),
  },
  onBroadcast: {},
});

// IMPORTANT: Export class name MUST be `Plugin`
export class Plugin extends BSBService, typeof EventSchemas> {
  static Config = Config;
  static EventSchemas = EventSchemas;

  constructor(cfg: BSBServiceConstructor>) {
    super(cfg);
  }

  async init(obs: Observable): Promise {
    // Listen for user.created events (with automatic validation)
    await this.events.onEvent('user.created', obs, async (childObs, data) => {
      await this.events.emitEvent('notification.send', childObs, {
        type: 'welcome',
        userId: data.userId,
        message: `Welcome ${data.email}!`,
      });
    });

    // Expose a returnable (RPC-like) handler (with automatic validation)
    await this.events.onReturnableEvent('api.getUser', obs, async (childObs, data) => {
      return { userId: data.userId, status: 'active' };
    });
  }

  async run(obs: Observable): Promise {
    await this.events.emitBroadcast('system.alert', obs, {
      level: 'info',
      message: 'My API service is now running',
      timestamp: new Date().toISOString(),
    });
  }

  async dispose() {}
}

Event Communication

BSB services communicate exclusively through typed events:

// Fire-and-forget events (single object parameter)
await this.events.emitEvent('notification.send', obs, {
  type: 'welcome',
  userId: 'user-123',
  message: 'Welcome!'
});

// Returnable events (RPC-style, single object parameter)
const result = await this.events.emitEventAndReturn('api.getUser', obs, {
  userId: 'user-123'
}, 5); // timeout in seconds

// Broadcast events (all listeners receive, single object parameter)
await this.events.emitBroadcast('system.alert', obs, {
  level: 'info',
  message: 'System status update',
  timestamp: new Date().toISOString()
});

// Listen to events (object parameter in handler)
await this.events.onEvent('user.created', obs, async (childObs, data) => {
  console.log(`New user: ${data.email}`);
});

// Expose returnable methods (object parameter and return)
await this.events.onReturnableEvent('api.getUser', obs, async (childObs, data) => {
  return { userId: data.userId, status: 'active' };
});

// Listen to broadcast events
await this.events.onBroadcast('system.maintenance', obs, async (childObs, data) => {
  console.log(`Maintenance starts at: ${data.startTime}`);
});

Event Name Uniqueness

Each event name belongs to one source service contract. Define the event on the plugin that owns/handles it, then consume that contract through the generated client from src/.bsb/clients. Do not copy the same event name into another plugin's emit* schema just because that plugin wants to call it.

Example: if service-tunnel-client-api exposes proxy.request in onReturnableEvents, service-tunnel-web should import the generated client and call it, not define its own emitReturnableEvents.proxy.request.

import { TunnelClientApiClient } from "../../.bsb/clients/service-tunnel-client-api.js";

export class Plugin extends BSBService<InstanceType<typeof Config>, typeof EventSchemas> {
  private readonly clientApi: TunnelClientApiClient;

  constructor(cfg: BSBServiceConstructor<InstanceType<typeof Config>>) {
    super(cfg);
    this.clientApi = new TunnelClientApiClient(this);
  }

  async run(obs: Observable) {
    const response = await this.clientApi.proxyRequest(obs, {
      hostname: "example.com",
      method: "GET",
      path: "/",
      headers: {},
    });
  }
}

Event Validation & Performance

BSB uses BSB types for event contracts and AnyVali for plugin config validation:

Observability First

Use the provided Observable for traces, spans, metrics, and logs. Do not create a separate logger for web servers, workers, queues, or helper modules. BSB logs are structured: the message is the stable template, and dynamic values belong in the metadata object.

async handleRequest(req: Request): Promise {
  const url = new URL(req.url);
  const requestObs = this.createTrace("http.request", {
    method: req.method,
    path: url.pathname,
  });

  requestObs.log.info("HTTP request started {method} {path}", {
    method: req.method,
    path: url.pathname,
  });

  try {
    const result = await this.clientApi.proxyRequest(requestObs, {
      hostname: "example.com",
      method: req.method,
      path: url.pathname,
      headers: {},
    });

    requestObs.log.info("HTTP request completed {status}", { status: result.status });
    requestObs.end({ status: result.status });
    return new Response(result.body, { status: result.status });
  } catch (error) {
    requestObs.log.error("HTTP request failed {error}", {
      error: error instanceof Error ? error.message : String(error),
    });
    requestObs.end({ error: true });
    throw error;
  }
}

Incoming work such as HTTP requests, queue messages, and scheduled jobs should create a new trace with this.createTrace(...). Nested local work should use child spans from the current observable. For BSB event calls, pass the current observable into the generated client or this.events; BSB already creates internal spans for event send/receive work, so do not double-span those calls.

Performance Considerations

  • Event schemas: Use BSB types for portable contracts and cross-language client generation
  • Config schemas: Use AnyVali for plugin configuration defaults and validation
  • Performance: Keep schemas focused and payloads bounded for predictable runtime cost
  • Portability: Exported schemas are consumed by registry tooling and generated clients
// Complex event schemas with nested objects and arrays
export const AdvancedEventSchemas = createEventSchemas({
  onEvents: {
    'order.process': createFireAndForgetEvent(
      bsb.object({
        id: bsb.uuid(),
        items: bsb.array(bsb.object({
          productId: bsb.uuid(),
          quantity: bsb.int32({ min: 1 }),
          price: bsb.number({ min: 0 })
        }, 'Order item'), { min: 1 }),
        customer: bsb.object({
          id: bsb.uuid(),
          email: bsb.string(),
          tier: bsb.enum(['basic', 'premium'])
        }, 'Customer'),
        metadata: optional(bsb.record(bsb.unknown()))
      }, 'Order payload'),
      'Process customer order with complex validation'
    )
  },
  
  emitReturnableEvents: {
    'payment.process': createReturnableEvent(
      bsb.object({
        orderId: bsb.uuid(),
        amount: bsb.number({ min: 0 }),
        currency: bsb.string({ min: 3, max: 3 }),
        paymentMethod: bsb.enum(['card', 'bank', 'crypto'])
      }, 'Payment request'),
      bsb.object({
        transactionId: bsb.string(),
        status: bsb.enum(['success', 'failed', 'pending']),
        fees: bsb.number({ min: 0 }),
        processedAt: bsb.datetime()
      }, 'Payment result'),
      'Process payment with detailed response'
    )
  }
});

// Use in your plugin class
export class AdvancedPlugin extends BSBService {
  static EventSchemas = AdvancedEventSchemas;

  constructor(cfg: BSBServiceConstructor) {
    super(cfg);
  }
}

Container Deployment

Deploy the BSB container and mount your prebuilt plugins:

docker run --rm \
  -v /path/to/plugins:/mnt/plugins:ro \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  betterweb/service-base:node
services:
  bsb:
    image: betterweb/service-base:node
    environment:
      - BSB_PLUGIN_DIRS=/mnt/plugins
      - BSB_PROFILE=default
    volumes:
      - ./plugins:/mnt/plugins:ro
    ports:
      - "8080:8080"  # if your service exposes HTTP
Important: Services communicate through the event bus and can run together in the same container or separately across multiple containers. The event bus handles all communication routing automatically.
