Production Deployment (BSB)

Goal:
- Deploy BSB service containers in production.
- Choose one config source, provide plugin packages, mount writable state explicitly, and keep build-time Docker concerns separate.
- For base image tags/env reference, use https://bsbcode.dev/llms/docker.txt
- For Node.js service image builds, use https://bsbcode.dev/llms/nodejs-docker.txt

Deployment model:
- Run the BSB runtime container.
- Provide plugins through mounted plugin directories or `BSB_PLUGINS`.
- Activate exactly one config plugin.
- Mount every stateful path explicitly.
- Prefer `code.bettercorp.dev/bettercorp/service-base:*` images in production. Docker Hub is a compatibility mirror.
- The official Node image includes `@bsb/base` built-in plugins: `config-default`, `config-env`, `events-default`, and `observable-default`.

What the `BSB_CONFIG_*` variables mean:
- `BSB_CONFIG_PLUGIN`: the active config plugin id. It decides where BSB reads runtime config from.
- `BSB_CONFIG_PLUGIN_PACKAGE`: the npm package that contains that config plugin.
- `BSB_CONFIG_FILE`: config file path used by `config-default`.
- `BSB_CONFIG_JSON`: full runtime config object used by `config-env`.
- `BSB_PROFILE`: profile name to select from profile-based config sources. Default is `default`.
- Config plugin schema fields are also read from env by exact, case-sensitive names. Example: Vault uses `vaultUrl`, not `VAULT_URL`.

YAML file config:
```yaml
services:
  app:
    image: code.bettercorp.dev/bettercorp/service-base:node
    environment:
      BSB_CONFIG_PLUGIN: config-default
      BSB_CONFIG_PLUGIN_PACKAGE: "@bsb/base"
      BSB_CONFIG_FILE: /app/sec-config.yaml
      BSB_PROFILE: default
    volumes:
      - ./sec-config.yaml:/app/sec-config.yaml:ro
      - ./plugins:/mnt/plugins:ro
```

Full JSON env config:
```yaml
services:
  app:
    image: code.bettercorp.dev/bettercorp/service-base:node
    environment:
      BSB_CONFIG_PLUGIN: config-env
      BSB_CONFIG_PLUGIN_PACKAGE: "@bsb/base"
      BSB_PROFILE: default
      BSB_CONFIG_JSON: >
        {"default":{"observable":{},"events":{},"services":{}}}
```

Vault runtime config:
```yaml
services:
  app:
    image: code.bettercorp.dev/bettercorp/service-base:node
    environment:
      BSB_CONFIG_PLUGIN: config-vault
      BSB_CONFIG_PLUGIN_PACKAGE: "@bsb/config-vault"
      vaultUrl: https://vault.example.com
      apiKeyId: vk_xxx
      apiSecret: vs_xxx
```

Vault rules:
- Runtime containers do not define application, service group/deployment, deployment profile, or config version in YAML/env.
- The Vault API key is bound server-side to those values.
- The container only receives the config plugin id/package and Vault API credentials.
- Do not add `@bsb/config-vault` to `BSB_PLUGINS` for the official Node image because it is bundled.

Writable paths:
```yaml
environment:
  BSB_WRITABLE_PATHS: /data,/var/lib/my-service/state.json
volumes:
  - app-data:/data
```
- BSB creates missing files/directories and fixes ownership/permissions before dropping to the runtime user.
- Use this for plugin state, uploads, SQLite/file DBs, caches, and generated key stores.

Startup plugin install:
```yaml
environment:
  BSB_PLUGINS: "@myorg/service-api@1.2.3,@myorg/observable-custom@2.4.1"
  BSB_PLUGIN_DIRS: /mnt/plugins
volumes:
  - bsb-plugins:/mnt/plugins
```
- Use exact versions where possible.
- If `BSB_PLUGINS` install fails, startup fails.
- Shared plugin storage uses package-scoped install locks, so one package install does not block unrelated packages.
- If image changes affected plugin install layout, run once with `BSB_PLUGIN_UPDATE=true` or clear the plugin cache volume.

Shared plugin watcher:
```yaml
services:
  plugin-watcher:
    image: code.bettercorp.dev/bettercorp/service-base:node
    environment:
      BSB_PLUGIN_WATCHER: "true"
      BSB_PLUGIN_DIRS: /mnt/plugins
      BSB_PLUGINS: "@myorg/service-api@1.2,@bsb/observable-axiom@9.6"
      BSB_PLUGIN_WATCH_INTERVAL_SECONDS: 3600
    volumes:
      - bsb-plugins:/mnt/plugins

  app:
    image: code.bettercorp.dev/bettercorp/service-base:node
    environment:
      BSB_PLUGIN_DIRS: /mnt/plugins
    volumes:
      - bsb-plugins:/mnt/plugins:ro
```
- Use watcher mode when many services share the same plugin storage.
- Runtime services should normally omit `BSB_PLUGINS`; they consume the synced plugin cache.
- Services pick up newly synced versions on restart.
- Exact selectors stay pinned. Minor/major/no selector allow controlled automatic updates according to npm resolution.

Vault manager service example:
```yaml
services:
  vault:
    image: code.bettercorp.dev/bettercorp/service-base:node-latest
    ports:
      - "3300:3300"
    environment:
      BSB_CONFIG_PLUGIN: config-env
      BSB_CONFIG_PLUGIN_PACKAGE: "@bsb/base"
      BSB_WRITABLE_PATHS: /data
      BSB_CONFIG_JSON: >
        {"default":{"observable":{},"events":{},"services":{"service-config-vault":{"package":"@bsb/config-vault","plugin":"service-config-vault","enabled":true,"config":{"host":"0.0.0.0","port":3300,"publicUrl":"https://vault.example.com","production":true,"databaseUrl":"postgres://bsb:bsb@postgres:5432/bsb_vault","masterKey":"BASE64_32_BYTE_KEY"}}}}}
    volumes:
      - vault-data:/data
    depends_on:
      - postgres

  postgres:
    image: postgres:18
    environment:
      POSTGRES_USER: bsb
      POSTGRES_PASSWORD: bsb
      POSTGRES_DB: bsb_vault
    volumes:
      - vault-pg:/var/lib/postgresql/data
```

Debugging startup:
- Set `BSB_SHOW_PACKAGES=true` to print discovered packages/plugins before BSB starts.
- The report shows package, version, plugin type, path, metadata path, and config reference snippets.
- Use `npm run plugins` inside the container for the same report.

Rules:
- Do not put runtime config or secrets into Dockerfiles.
- Do not run plain `tsc`, `node`, `tsx`, or `ts-node` as the service build path; plugin builds must use the BSB plugin lifecycle.
- Do not compile plugin source in production containers.
- Use `/guides/nodejs/docker/` only for building an image; use this guide for deployment.
