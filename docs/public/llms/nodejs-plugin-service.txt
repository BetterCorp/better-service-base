Create A New Service (Node.js)

Goal:
- Add a service plugin in src/plugins/service-<name>/.
- Node.js plugins are ESM-only. Use `"type": "module"`, `NodeNext`, and `.js` specifiers for relative imports.

Steps:
1. Create plugin folder: src/plugins/service-<name>.
2. Create index.ts with Config + EventSchemas + Plugin (BSBService).
   - Config defines plugin metadata and config schema (only include fields you actually have).
   - EventSchemas defines emitted/listened events (required for clients and docs).
   - Every EventSchemas key must be unique across emitEvents, onEvents, emitReturnableEvents, onReturnableEvents, emitBroadcast, and onBroadcast.
   - Event names belong to the plugin that owns/handles that service contract. Do not duplicate the same event name in another plugin's emit* schema just because that plugin wants to call it.
   - If this plugin needs to call/listen to another service contract, import that service's generated client from `../../.bsb/clients/<plugin-id>.js`, instantiate it in the constructor, then call the generated method.
   - Example: if service-tunnel-client-api exposes `proxy.request` in onReturnableEvents, service-tunnel-web should use `new TunnelClientApiClient(this)` and `clientApi.proxyRequest(...)`, not define `emitReturnableEvents["proxy.request"]`.
   - Plugin implements lifecycle: init(obs), run(obs), dispose().
   - this.config is fully typed from the config schema (via BSBService<InstanceType<typeof Config>, ...>). Never cast it.
   - Do not pass `{ unknownKeys: "strip" }` to plugin config `av.object(...)`; BSB enforces `schema.parse(config, { unknownKeys: "strip" })` centrally at startup.
   - For config fields with defaults use av.string().default("x") (key optional in config file, type stays string).
     Only use av.optional(...) for fields with no default that may be absent (type becomes T | undefined).

3. Update README.md and docs/ (registry docs).
4. Ensure bsb-plugin.json exists (generated by bsb-plugin-cli build; do not edit manually).
5. Add bsb-tests.json entry when you need test config (optional, not user-facing).

Service expectations:
- constructor - sets up class properties and/or other BSB client plugins.
- init(obs) - sets up resources and all onEvents.
- run(obs) - starts the service (long-lived loop or server).
- dispose() - cleans up.

Observability rules:
- Use the provided `Observable` everywhere. Do not create a separate logger for web servers, queues, workers, or helpers.
- Incoming work (HTTP requests, queue messages, scheduled jobs) should start a root trace: `const requestObs = this.createTrace("http.request", { method, path })`.
- Nested work should create child spans from the current observable: `const dbObs = requestObs.startSpan("db.query", { table: "users" })`.
- Pass the current/child observable into downstream generated client calls, DB helpers, queues, and nested work.
- Do not manually wrap BSB event calls in extra spans. BSB already creates internal spans around event send/receive work; just pass the current observable into generated clients or `this.events`.
- Logs must use structured metadata: `obs.log.info("HTTP request completed {status}", { status })`. Do not interpolate dynamic values into plain strings and do not log without metadata when data matters.
- On errors, log structured error data and end the span with error metadata before rethrowing or returning the error response.

Tests:
- Add custom tests in tests/service-<name>/.
