Docker Images (BSB)

Goal:
- Explain official BSB Docker images, image tags, base image defaults, startup behavior, and container environment variables.
- Keep this separate from production deployment recipes and language-specific service builds.
- For production deployment, use https://bsbcode.dev/llms/deploy-production.txt
- For Node.js Docker builds, use https://bsbcode.dev/llms/nodejs-docker.txt

Image tags:
- Primary registry: `code.bettercorp.dev/bettercorp/service-base`.
- Docker Hub mirror: `betterweb/service-base`. Treat it as best-effort fallback because Docker Hub rate limits can delay or skip mirror pushes.
- Node.js runtime: `code.bettercorp.dev/bettercorp/service-base:node` or `code.bettercorp.dev/bettercorp/service-base:node-latest`.
- Version-specific Node.js runtime: `code.bettercorp.dev/bettercorp/service-base:node-<version>`, for example `code.bettercorp.dev/bettercorp/service-base:node-9.6.46`.
- Do not use tags like `9-node`; BSB tags are runtime-first, not version-first.
- Other runtime tags follow the same pattern: `go`, `go-latest`, `go-<version>`, `python`, `python-latest`, `dotnet`, etc.

Node image contents:
- BSB runs from `/home/bsb/node_modules/@bsb/base`, not from a copied local `/home/bsb/lib` tree.
- Built-in plugins resolve from `@bsb/base`.
- The official Node image includes `@bsb/config-vault` in `/home/bsb/node_modules` when built by the official release workflow.
- Reference bundled Vault plugins with `package: "@bsb/config-vault"` or `BSB_CONFIG_PLUGIN_PACKAGE=@bsb/config-vault`.
- Do not add bundled packages such as `@bsb/config-vault` to `BSB_PLUGINS`; that forces startup install work.
- The runtime package exposes useful shell commands: `npm run start`, `npm run debug`, and `npm run plugins`.
- The same image can run plugin watcher mode with `BSB_PLUGIN_WATCHER=true`; watcher mode syncs plugins and never starts BSB.

Default environment:
- The official Node image already sets:
  - `NODE_ENV=production`
  - `BSB_CONTAINER=true`
  - `BSB_LIVE=true`
  - `BSB_PLUGIN_DIRS=/mnt/plugins`
- Dockerfiles that extend `code.bettercorp.dev/bettercorp/service-base:node` should not restate these defaults unless intentionally overriding them.

Entrypoint flow:
- Print the BSB runtime version.
- Prepare writable paths from `BSB_WRITABLE_PATHS`.
- Ensure plugin storage exists.
- Install packages listed in `BSB_PLUGINS`, if any.
- Optionally print package/plugin discovery when `BSB_SHOW_PACKAGES=true`.
- Start BSB.
- If requested plugin installation fails, the container must exit instead of starting without the requested plugins.

Plugin loading environment:
- `BSB_PLUGIN_DIRS`: comma-separated plugin directories. Prefer this over legacy aliases.
- `BSB_PLUGIN_DIR` and `BSB_PLUGINS_DIR`: legacy aliases; still work and accept comma-separated values.
- `BSB_PLUGINS`: comma-separated npm packages installed at container start.
- Supported `BSB_PLUGINS` selectors: no selector, major, minor, or exact version.
- `@latest` is rejected; omit the selector for npm latest.
- Prefer exact versions in production. Use major/minor selectors only when controlled updates are intended.
- `BSB_PLUGIN_UPDATE=true`: refresh already-installed plugins on startup.
- `BSB_PLUGIN_WATCHER=true`: run plugin watcher mode instead of BSB.
- `BSB_PLUGIN_WATCH_INTERVAL_SECONDS`: watcher image sync interval. Default: 3600.
- `BSB_PLUGIN_SYNC_INTERVAL_SECONDS`: legacy/alias interval name for watcher mode.
- `BSB_PLUGIN_WATCH_ONCE=true`: watcher mode runs one sync and exits.

Config selection environment:
- `BSB_CONFIG_PLUGIN`: active config plugin id. Common values: `config-default`, `config-env`, `config-vault`.
- `BSB_CONFIG_PLUGIN_PACKAGE`: npm package containing the config plugin.
- Use `@bsb/base` for `config-default` and `config-env`.
- Use `@bsb/config-vault` for `config-vault`.
- Exactly one config plugin is active at runtime. BSB does not activate multiple config plugins as a fallback chain.
- Config plugin schema fields are read from env by exact, case-sensitive key names.

Common config plugin schema values:
- `config-default`: uses `BSB_CONFIG_FILE` for the YAML config path.
- `config-env`: uses `BSB_CONFIG_JSON` for the full runtime config object.
- `config-vault`: uses `vaultUrl`, `apiKeyId`, `apiSecret`, and optional fields such as `timeoutMs` and `allowInsecureHttp`.
- `BSB_PROFILE`: selects the deployment profile in config sources that expose profiles, default `default`.

Other runtime environment:
- `BSB_WRITABLE_PATHS`: comma-separated files/directories that the Node entrypoint creates if needed and makes writable for runtime state, for example `/data`.
- `BSB_DEBUG=true`: enable production debug logging.
- `BSB_SHOW_PACKAGES=true`: print a startup package/plugin discovery report, including package names, versions, plugin types, paths, and config-reference snippets.

Plugin search report:
- Runtime: set `BSB_SHOW_PACKAGES=true`.
- Inside a running/debug shell: `npm run plugins`.
- In a derived Dockerfile after installing/copying plugins:
  `RUN node /home/bsb/node_modules/@bsb/base/lib/scripts/list-plugin-search-paths.js`
- In the BSB repo itself:
  `npm run list-plugin-search-paths --workspace nodejs`

Rules:
- Do not compile plugin source in the production container.
- Do not invent Docker tags; use runtime-prefixed tags.
- Prefer Forgejo image names in production. Docker Hub image names are compatibility mirrors.
- Keep deployment config out of service build Dockerfiles.
