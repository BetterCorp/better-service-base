Docker (BSB)

Goal:
- Run BSB in production with the official Docker image and mounted/installed plugins.

Image tags:
- Node.js runtime: `betterweb/service-base:node` or `betterweb/service-base:node-latest`.
- Version-specific Node.js runtime: `betterweb/service-base:node-<version>`, for example `betterweb/service-base:node-9.5.1`.
- Do NOT use tags like `9-node`; BSB tags are runtime-first, not version-first.
- Other runtime tags follow the same pattern: `go`, `go-latest`, `go-<version>`, `python`, `python-latest`, `dotnet`, etc.

Mounted plugin deployment:
```bash
docker run --rm \
  -v "$(pwd)/plugins:/mnt/plugins:ro" \
  -v "$(pwd)/sec-config.yaml:/app/sec-config.yaml:ro" \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  -e BSB_CONFIG_FILE=/app/sec-config.yaml \
  betterweb/service-base:node
```

Install plugins at startup:
```bash
docker run --rm \
  -v bsb-plugins:/mnt/plugins \
  -v "$(pwd)/sec-config.yaml:/app/sec-config.yaml:ro" \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  -e BSB_PLUGINS='@myorg/service-api@1.2.3,@myorg/observable-custom@latest' \
  -e BSB_CONFIG_FILE=/app/sec-config.yaml \
  betterweb/service-base:node-latest
```

Compose:
```yaml
services:
  bsb:
    image: betterweb/service-base:node
    environment:
      BSB_PLUGIN_DIRS: /mnt/plugins
      BSB_CONFIG_FILE: /app/sec-config.yaml
    volumes:
      - ./plugins:/mnt/plugins:ro
      - ./sec-config.yaml:/app/sec-config.yaml:ro
```

Environment:
- `BSB_PLUGIN_DIRS`: comma-separated plugin directories. Prefer this over legacy aliases.
- `BSB_PLUGIN_DIR` and `BSB_PLUGINS_DIR`: legacy aliases; still work and accept comma-separated values.
- `BSB_PLUGINS`: comma-separated npm packages installed at container start.
- `BSB_PLUGIN_UPDATE=true`: refresh already-installed plugins on startup.
- `BSB_CONFIG_FILE`: config file path inside the container.
- `BSB_PROFILE`: config profile to load, default `default`.
- `BSB_DEBUG=true`: enable production debug logging.

Rules:
- Build plugin packages before deployment with `bsb-plugin-cli build`.
- Do not compile plugin source in the production container.
- Do not invent Docker tags; use the runtime-prefixed tags above.
