Docker (BSB)

Goal:
- Run BSB in production with the official Docker image and mounted/installed plugins.

Image tags:
- Node.js runtime: `betterweb/service-base:node` or `betterweb/service-base:node-latest`.
- Version-specific Node.js runtime: `betterweb/service-base:node-<version>`, for example `betterweb/service-base:node-9.5.1`.
- Do NOT use tags like `9-node`; BSB tags are runtime-first, not version-first.
- Other runtime tags follow the same pattern: `go`, `go-latest`, `go-<version>`, `python`, `python-latest`, `dotnet`, etc.

Bundled Node.js plugins:
- The Node.js runtime image includes `@bsb/config-vault` in `/home/bsb/node_modules` when built by the official release workflow.
- Reference bundled Vault plugins by package name in config: `package: "@bsb/config-vault"`.
- Do not set `BSB_PLUGINS` for bundled plugins such as `@bsb/config-vault`; that forces the startup installer and can slow or break boot.
- Other plugins should still be mounted with `BSB_PLUGIN_DIRS` or, when needed, installed at startup with pinned `BSB_PLUGINS` versions.

Mounted plugin deployment:
```bash
docker run --rm \
  -v "$(pwd)/plugins:/mnt/plugins:ro" \
  -v "$(pwd)/sec-config.yaml:/app/sec-config.yaml:ro" \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  -e BSB_CONFIG_FILE=/app/sec-config.yaml \
  betterweb/service-base:node
```

Install plugins at startup:
```bash
docker run --rm \
  -v bsb-plugins:/mnt/plugins \
  -v "$(pwd)/sec-config.yaml:/app/sec-config.yaml:ro" \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  -e BSB_PLUGINS='@myorg/service-api@1.2.3,@myorg/observable-custom@2.4.1' \
  -e BSB_CONFIG_FILE=/app/sec-config.yaml \
  betterweb/service-base:node-latest
```

Compose:
```yaml
services:
  bsb:
    image: betterweb/service-base:node
    environment:
      BSB_PLUGIN_DIRS: /mnt/plugins
      BSB_CONFIG_FILE: /app/sec-config.yaml
      BSB_WRITABLE_PATHS: /data
    volumes:
      - ./plugins:/mnt/plugins:ro
      - ./sec-config.yaml:/app/sec-config.yaml:ro
      - bsb-data:/data
```

Environment:
- The official Node image already sets `NODE_ENV=production`, `BSB_CONTAINER=true`, `BSB_LIVE=true`, and `BSB_PLUGIN_DIRS=/mnt/plugins`.
- Dockerfiles that extend `betterweb/service-base:node` or `betterweb/service-base:node-latest` should not restate those defaults unless they intentionally need to override them.
- `BSB_PLUGIN_DIRS`: comma-separated plugin directories. Prefer this over legacy aliases.
- `BSB_PLUGIN_DIR` and `BSB_PLUGINS_DIR`: legacy aliases; still work and accept comma-separated values.
- `BSB_PLUGINS`: comma-separated npm packages installed at container start. Prefer exact versions; avoid unversioned packages and `@latest`.
- `BSB_PLUGIN_UPDATE=true`: refresh already-installed plugins on startup.
- `BSB_WRITABLE_PATHS`: comma-separated files/directories that the Node entrypoint creates if needed and makes writable for runtime state, for example `/data`.
- `BSB_CONFIG_FILE`: config file path inside the container.
- `BSB_CONFIG_PLUGIN=config-env` and `BSB_CONFIG_JSON=...`: use these instead of `BSB_CONFIG_FILE` when the full runtime config is supplied by environment.
- `BSB_PROFILE`: config profile to load, default `default`.
- `BSB_DEBUG=true`: enable production debug logging.

Rules:
- Do not add `@bsb/config-vault` to `BSB_PLUGINS` when using an official Node image that already bundles it.
- Build plugin packages before deployment with `bsb-plugin-cli build`.
- Do not compile plugin source in the production container.
- Do not invent Docker tags; use the runtime-prefixed tags above.
