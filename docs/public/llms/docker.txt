Docker (BSB)

Goal:
- Run BSB in production with the official Docker image and mounted/installed plugins.

Image tags:
- Primary registry: `code.bettercorp.dev/bettercorp/service-base`.
- Docker Hub mirror: `betterweb/service-base`. Treat it as best-effort fallback because Docker Hub rate limits can delay or skip mirror pushes.
- Node.js runtime: `code.bettercorp.dev/bettercorp/service-base:node` or `code.bettercorp.dev/bettercorp/service-base:node-latest`.
- Version-specific Node.js runtime: `code.bettercorp.dev/bettercorp/service-base:node-<version>`, for example `code.bettercorp.dev/bettercorp/service-base:node-9.5.1`.
- Do NOT use tags like `9-node`; BSB tags are runtime-first, not version-first.
- Other runtime tags follow the same pattern: `go`, `go-latest`, `go-<version>`, `python`, `python-latest`, `dotnet`, etc.

Bundled Node.js plugins:
- The Node.js runtime image runs BSB from `/home/bsb/node_modules/@bsb/base`, not from a copied local `/home/bsb/lib` tree.
- Built-in core plugins resolve from `@bsb/base`; plugin configs should use package-qualified plugins for non-local packages.
- The runtime package includes `npm run start`, `npm run debug`, and `npm run plugins` for shell/debug use inside the container.
- The Node.js runtime image includes `@bsb/config-vault` in `/home/bsb/node_modules` when built by the official release workflow.
- Reference bundled Vault plugins by package name in config: `package: "@bsb/config-vault"`.
- Do not set `BSB_PLUGINS` for bundled plugins such as `@bsb/config-vault`; that forces the startup installer and can slow or break boot.
- Other plugins should still be mounted with `BSB_PLUGIN_DIRS` or, when needed, installed at startup with pinned `BSB_PLUGINS` versions.

Mounted plugin deployment:
```bash
docker run --rm \
  -v "$(pwd)/plugins:/mnt/plugins:ro" \
  -v "$(pwd)/sec-config.yaml:/app/sec-config.yaml:ro" \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  -e BSB_CONFIG_FILE=/app/sec-config.yaml \
  code.bettercorp.dev/bettercorp/service-base:node
```

Install plugins at startup:
```bash
docker run --rm \
  -v bsb-plugins:/mnt/plugins \
  -v "$(pwd)/sec-config.yaml:/app/sec-config.yaml:ro" \
  -e BSB_PLUGIN_DIRS=/mnt/plugins \
  -e BSB_PLUGINS='@myorg/service-api@1.2.3,@myorg/observable-custom@2.4.1' \
  -e BSB_CONFIG_FILE=/app/sec-config.yaml \
  code.bettercorp.dev/bettercorp/service-base:node-latest
```

Compose:
```yaml
services:
  bsb:
    image: code.bettercorp.dev/bettercorp/service-base:node
    environment:
      BSB_PLUGIN_DIRS: /mnt/plugins
      BSB_CONFIG_FILE: /app/sec-config.yaml
      BSB_WRITABLE_PATHS: /data
    volumes:
      - ./plugins:/mnt/plugins:ro
      - ./sec-config.yaml:/app/sec-config.yaml:ro
      - bsb-data:/data
```

Environment:
- The official Node image already sets `NODE_ENV=production`, `BSB_CONTAINER=true`, `BSB_LIVE=true`, and `BSB_PLUGIN_DIRS=/mnt/plugins`.
- Dockerfiles that extend `code.bettercorp.dev/bettercorp/service-base:node` or `code.bettercorp.dev/bettercorp/service-base:node-latest` should not restate those defaults unless they intentionally need to override them.
- `BSB_PLUGIN_DIRS`: comma-separated plugin directories. Prefer this over legacy aliases.
- `BSB_PLUGIN_DIR` and `BSB_PLUGINS_DIR`: legacy aliases; still work and accept comma-separated values.
- `BSB_PLUGINS`: comma-separated npm packages installed at container start. Supported selectors are no selector, `@X`/`:X`, `@X.Y`/`:X.Y`, or exact `@X.Y.Z`/`:X.Y.Z`. Prefer exact versions for production; use major/minor selectors only when you intentionally want controlled updates. `@latest` is rejected; use no selector if you want npm latest.
- `BSB_PLUGIN_UPDATE=true`: refresh already-installed plugins on startup.
- `BSB_WRITABLE_PATHS`: comma-separated files/directories that the Node entrypoint creates if needed and makes writable for runtime state, for example `/data`.
- `BSB_CONFIG_FILE`: config file path inside the container.
- `BSB_CONFIG_PLUGIN=config-env` and `BSB_CONFIG_JSON=...`: use these instead of `BSB_CONFIG_FILE` when the full runtime config is supplied by environment.
- `BSB_PROFILE`: config profile to load, default `default`.
- `BSB_DEBUG=true`: enable production debug logging.
- `BSB_SHOW_PACKAGES=true`: print a startup package/plugin discovery report, including package names, versions, plugin types, paths, and config-reference snippets.

Rules:
- Do not add `@bsb/config-vault` to `BSB_PLUGINS` when using an official Node image that already bundles it.
- Build plugin packages before deployment with `bsb-plugin-cli build`.
- If `BSB_PLUGINS` is set and install/validation fails, the container must exit instead of starting BSB without the requested plugins.
- After changing BSB image versions that affect plugin install layout, run once with `BSB_PLUGIN_UPDATE=true` or clear the plugin cache volume so stale `/mnt/plugins/<package>/<version>` installs are rebuilt.
- To log what plugins an image can see during a derived Docker build, run `node /home/bsb/node_modules/@bsb/base/lib/scripts/list-plugin-search-paths.js` after installing/copying plugins. At runtime, set `BSB_SHOW_PACKAGES=true` for the same report before BSB starts.
- In the BSB repo itself, the equivalent command is `npm run list-plugin-search-paths --workspace nodejs`; it runs automatically during Node build/build-release.
- Do not compile plugin source in the production container.
- Do not invent Docker tags; use the runtime-prefixed tags above.
- Prefer Forgejo image names in production. Docker Hub image names are compatibility mirrors.
