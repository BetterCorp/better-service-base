Node.js Bootstrap (BSB)

Goal:
- Create a minimal BSB Node.js project with plugins in src/plugins/* and a working runtime.

Prereqs:
- Node >= 23, npm >= 11.
- This project uses @bsb/base at runtime (framework + plugin CLI).
- Node.js plugins are ESM-only. Use `"type": "module"` in `package.json`, `module: "NodeNext"` in `tsconfig.json`, and `.js` specifiers for relative imports in TypeScript source.

Bootstrap steps:
1. Create a new npm project (npm init) and install `@bsb/base` as a dev and peer dependency.
2. Make plugins directory `src/plugins/`
3. Add `/tsconfig.json` with:
```json
{
  "compilerOptions": {
    "target": "ES2022", // (or newer)
    "module": "NodeNext",
    "lib": ["ES2022"],
    "declaration": true,
    "declarationMap": true,
    "sourceMap": true,
    "outDir": "./lib",
    "rootDir": "./src",
    // Strict type checking - NO `any` types allowed!
    // BSB is designed to be fully type-safe across languages
    "strict": true, // Includes noImplicitAny
    "noImplicitAny": true, // Explicit: error on implicit any
    "esModuleInterop": true,
    "skipLibCheck": true,
    "forceConsistentCasingInFileNames": true,
    "moduleResolution": "NodeNext",
    "resolveJsonModule": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```
4. Add a default sec-config.yaml:
Note: config plugins are not defined here as they load the config (this file is loaded by the config-default). If you use a remote config plugin (e.g., config-vault), sec-config.yaml may not be used.
```yaml
default: # this is the deployment profile
  observable:
    observable-default:
      plugin: observable-default
      enabled: true
      config: {}
  events:
    events-default:                    # Use events-default for local, or events-rabbitmq for distributed
      plugin: events-default
      enabled: true
  services: {} # or actual plugins list
    # {mapped plugin name}:
    #   plugin: {actual plugin name}
    #   # package: "{npm package}" # Uncomment if not running in CWD
    #   enabled: true
    #   config: {} # Or actual config

```
5. Add the following to package.json:
```json 
{
  "type": "module",
  "files": [
    "lib/**/*",
    "README.md",
    "bsb-plugin.json",
    "bsb-tests.json"
  ],
  "scripts": {
    "build": "bsb-plugin-cli build",
    "clean": "bsb-plugin-cli clean",
    "start": "bsb-plugin-cli start",
    "dev": "bsb-plugin-cli dev",
    "test": "bsb-plugin-cli test",
    "publish:client": "bsb client publish"
  },
  "bsb": {}
}
```
Make sure there is a license defined otherwise default to: `(AGPL-3.0-only OR Commercial)`

Notes:
- bsb-plugin.json is generated by bsb-plugin-cli build; do not edit manually.
- bsb-tests.json is optional and used only for test config (not user-facing).
- Author relative TypeScript imports with `.js` specifiers, for example `import { helper } from "./helper.js";`.
- `npm run dev` is the development entrypoint and can load `.ts` plugins. `npm start` is the production entrypoint and loads compiled `.js` only.
- `npm run build` must run `bsb-plugin-cli build`, not `tsc`. BSB build runs hooks, schema extraction, virtual client generation, TypeScript, asset/schema copying, and manifests.
- Do not add scripts that run plugin files directly with `node`, `tsx`, or `ts-node`. Runtime code should enter through BSB (`bsb-plugin-cli dev`, `bsb-plugin-cli start`, tests, or hooks) so lifecycle, config, events, and observability are present.
- Keep each plugin self-contained: place plugin schemas, assets, helpers, generated inputs, and local docs under the plugin project/folder. Reference them from elsewhere when needed; do not make plugins depend on shared side directories.

Recommended defaults (for dev):
Config: config-default.
Observable: observable-default.
Events: events-default.

(change plugins for production deployment - get plugins from BSB Registry for more information - https://io.bsbcode.dev/llms.txt)

Container deployment:
- BSB is designed to run as the `code.bettercorp.dev/bettercorp/service-base:node` Docker image.
- Use `code.bettercorp.dev/bettercorp/service-base:node` or `code.bettercorp.dev/bettercorp/service-base:node-latest`; version-specific tags are `node-<version>` like `node-9.5.1`. Never use `9-node`.
- Docker Hub `betterweb/service-base` tags are compatibility mirrors, not the primary image source.
- Provide plugins via `BSB_PLUGIN_DIRS` env var (comma-separated paths). First dir is used as install target.
- `BSB_PLUGIN_DIR` and `BSB_PLUGINS_DIR` are legacy aliases (still work, accept comma-separated).
- `BSB_PLUGINS` env var: comma-separated npm packages installed at container start (e.g. `@scope/plugin@1.2.3`).
- `BSB_PLUGIN_UPDATE=true`: refresh already-installed plugins on startup.
- `BSB_DEBUG=true`: enable debug logging in production mode (mode becomes `production-debug`).
- `NODE_ENV=production`, `BSB_CONTAINER=true`, `BSB_LIVE=true`, and `BSB_PLUGIN_DIRS=/mnt/plugins` are set by the base image. Dockerfiles that extend the BSB Node image do not need to set these again unless intentionally overriding them.
- To log plugin discovery during a derived container build, run `node /home/bsb/lib/scripts/list-plugin-search-paths.js` after plugins have been copied or installed. In the BSB repo, run `npm run list-plugin-search-paths --workspace nodejs`; the core Node build runs it automatically.
- Plugin dirs are read/search roots. Do not require BSB to write `package.json` into them; versioned plugin package directories contain package metadata.
- Full Docker guide: https://bsbcode.dev/llms/docker.txt

Build hooks (optional):
- To inject custom scripts (codegen, transforms, packaging) into the build/dev pipeline, configure `bsb.hooks` in package.json.
- See https://bsbcode.dev/llms/nodejs-build-hooks.txt
